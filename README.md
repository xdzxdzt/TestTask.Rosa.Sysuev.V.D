# TestTask.Rosa.Sysuev.V.D

Web API для учета заявок сотрудников на бухгалтерские справки.

## Описание задачи

Сервис автоматизирует процесс подачи и обработки заявок на бухгалтерские справки. Сотрудник может создать заявку и отслеживать ее статус, а бухгалтер может просматривать заявки и менять их статус в процессе обработки.

Поддерживаемые типы справок:

- `TwoNDFL` - справка 2-НДФЛ;
- `PlaceOfWorkAndExperience` - справка о месте работы и стаже;
- `AverageSalary` - справка о среднем заработке;
- `Other` - справка произвольного типа.

Статусы заявки:

- `Created` - заявка создана;
- `InProgress` - заявка находится в работе;
- `Closed` - заявка закрыта.

## Реализованная функциональность

- Создание заявки сотрудником.
- Получение всех заявок.
- Получение заявки по идентификатору.
- Получение заявок конкретного сотрудника.
- Изменение статуса заявки бухгалтером.
- Проверка, что заявку может создать только пользователь с ролью `Employee`.
- Проверка, что статус может менять только пользователь с ролью `Accountant`.
- Защита от повторной активной заявки одного типа для одного сотрудника.
- Валидация доменной модели заявки: пустые идентификаторы, количество экземпляров, причина, даты закрытия.

## Требования для запуска

- .NET SDK 8.0
- PostgreSQL

Строка подключения находится в:

```text
TestTask.Rosa.API/appsettings.json
```

По умолчанию используется:

```json
"Database": "Host=localhost;Port=5432;Database=TestTaskRosaDb;Username=postgres;Password=root"
```

При необходимости измените `Username`, `Password`, `Host` и `Port` под свою локальную БД.

## Запуск проекта

1. Восстановить зависимости:

```bash
dotnet restore
```

2. Применить миграции:

```bash
dotnet ef database update --project TestTask.Rosa.DataAccess --startup-project TestTask.Rosa.API
```

3. Запустить API:

```bash
dotnet run --project TestTask.Rosa.API
```

4. Открыть Swagger:

```text
https://localhost:<port>/swagger
```

Точный порт можно посмотреть в:

```text
TestTask.Rosa.API/Properties/launchSettings.json
```

## API

### Создать заявку

```http
POST /api/reference/create
```

Пример тела запроса:

```json
{
  "employeeId": "00000000-0000-0000-0000-000000000000",
  "referenceType": "TwoNDFL",
  "copiesCount": 1,
  "reason": "Для предоставления по месту требования"
}
```

Успешный ответ:

```json
{
  "id": "00000000-0000-0000-0000-000000000000"
}
```

### Получить все заявки

```http
GET /api/reference/get-all
```

Ответ содержит список заявок с данными сотрудника:

```json
[
  {
    "id": "00000000-0000-0000-0000-000000000000",
    "employeeId": "00000000-0000-0000-0000-000000000000",
    "firstNameEmployee": "Иван",
    "lastNameEmployee": "Иванов",
    "type": "TwoNDFL",
    "copiesCount": 1,
    "reason": "Для предоставления по месту требования",
    "status": "Created",
    "createdAt": "2026-05-25T10:00:00Z",
    "updatedAt": null,
    "closedAt": null
  }
]
```

### Получить заявку по идентификатору

```http
GET /api/reference/{id}
```

### Получить заявки сотрудника

```http
GET /api/reference/employee/{employeeId}
```

### Обновить статус заявки

```http
PATCH /api/reference/{id}/status
```

Пример тела запроса:

```json
{
  "accountantId": "00000000-0000-0000-0000-000000000000",
  "status": "InProgress"
}
```

## Бизнес-правила

- Создавать заявки может только пользователь с ролью `Employee`.
- Изменять статус заявки может только пользователь с ролью `Accountant`.
- У одного сотрудника не может быть двух активных заявок одного типа.
- Количество экземпляров справки должно быть больше нуля.
- Закрытая заявка не может быть изменена.

## Обработка ошибок

Ошибки бизнес-логики возвращаются через `BadRequest` или `NotFound` с текстом причины.

Примеры:

- сотрудник не найден;
- создавать заявки может только сотрудник;
- изменять статус может только бухгалтер;
- справка такого типа уже в работе;
- справка не найдена;
- закрытую заявку нельзя изменить.

Непредвиденные ошибки возвращаются как:

```http
500 Internal Server Error
```

## Ответы на вопросы для рефлексии

### 1. Какое архитектурное решение вы считаете самым рискованным и почему?

Самым рискованным решением является проверка повторной активной заявки только на уровне сервисной логики. Сейчас сервис проверяет наличие активной заявки перед созданием новой, но при параллельных запросах возможна гонка: два запроса могут одновременно пройти проверку и создать дубликаты.

### 2. Где вы сознательно оставили технический долг? Как бы его закрыли при наличии времени?

Технический долг:

- нет отдельного middleware для централизованной обработки ошибок;
- нет логирования исключений;
- нет пагинации для получения всех заявок;
- строка подключения хранится в `appsettings.json`;
- нет автоматических тестов для бизнес-правил.

### 3. Что бы вы изменили, если бы знали, что сервис станет высоконагруженным?

Для высоконагруженного сценария я бы изменил следующее:

- добавил пагинацию и фильтры для очереди заявок;
- добавил индексы по `UserId`, `Status`, `Type`, `CreatedAt`;
